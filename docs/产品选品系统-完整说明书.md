## 1. 系统概述

### 1.1 这是什么

一个 Amazon 精铺选品的**半自动化分析系统**。你给一个 ASIN 或关键词，它用 Sorftime CLI 采集市场数据，通过三层决策体系（市场/财务/资金）判断能不能做，结果写入飞书 6 张表供长期追踪。

### 1.2 核心能力

| 能力 | 说明 |
|------|------|
| **3 入口** | ASIN / 关键词 / 类目 NodeId |
| **3 层决策** | Market（市场好不好）→ Financial（能不能赚钱）→ Launch（资金撑不撑得住） |
| **属性标注+交叉分析** | Top100 产品按维度标注，交叉矩阵发现空白供给 |
| **反向财务** | 没给成本时自动算出"采购价必须低于多少才能赚钱" |
| **飞书 6 表闭环** | 分析数据持久化，可追溯，可复盘 |
| **后验追踪** | 上线后标记成功/失败，积累"什么样的产品在我手上真的成功过" |

### 1.3 不做什么

- 不自动发现新产品方向（你需要自己给 ASIN 或关键词）
- 不操作广告/Listing/库存
- 不替代人工决策（专利/合规/供应链仍需你确认）

### 1.4 技术架构

```
用户输入(ASIN/关键词)
    ↓
Sorftime CLI (43个Amazon端点) → 原始数据采集
    ↓
Claude AI (编排+解释) → 三层决策 + 属性标注 + 交叉分析 + VOC
    ↓
确定性脚本 (8个Python + 1个Node.js) → 评分计算 + 财务引擎 + 报告lint
    ↓
飞书 MCP → 6表写入 + 回读验证
    ↓
磁盘 → decision-card.html + full-report.md
```

## 2. 第一性原理

### 2.1 核心信念

**选择比努力重要，但选择不能用猜的。**

精铺卖家每一款产品都要真金白银投进去。选错了不是浪费一次分析——是浪费 $3,000-15,000 的首批备货 + 3-6 个月的运营时间。

### 2.2 三个问题

| 问题 | 这个系统做什么 |
|------|---------------|
| **能不能做？** | 三层决策——市场说了能追、财务说了能赚、资金说了能撑，三个都过才过 |
| **值得做吗？** | 财务反向测算——没给成本时，告诉你"采购必须低于多少才能赚钱"。不给幻觉，给数字 |
| **做了之后怎么变聪明？** | 飞书 6 表 + 后验追踪——积累"什么样的产品在我们手里真的成功过" |

### 2.3 精铺模式适配

- **利润敏感**：M≥2.0 且净利率≥15% 才判 GO。即使在 50% 广告占比下也要保证盈利
- **现金流意识**：反向财务告诉你成本上限。现金流撑不到回本 = 不投
- **长期沉淀**：后验追踪字段让每次失败都变成可复用的判断规则

## 3. 快速开始

### 3.1 最简用法——给一个 ASIN

```
帮我分析 B0DH2NGXNB
```

系统自动执行：产品详情 → 关键词 → 类目 Top100 → 属性标注 → 三层决策 → 飞书写入

### 3.2 从关键词进入——市场初筛

```
printer stand 这个品类值得做吗
```

系统从关键词出发定位类目，做市场级分析。只能做反向财务（成本上限），不能做精确 P&L。

### 3.3 中途给成本

```
成本 $3.2，头程 $0.8，CPC $0.85，CVR 9%
```

系统收到后自动触发财务分析，计算 M 值、净利率、回本周期等 48 个字段。

### 3.4 批量模式

```
产品初选表里待分析的都帮我跑一遍
```

遍历飞书候选队列，逐个分析。

## 4. 完整使用路径

### 4.1 路径 A：从 ASIN 进入（完整路径）

```
ASIN (如 B0DH2NGXNB)
    │
    ├─→ ProductRequest → 精确售价/FBA/佣金/利润率
    ├─→ BsrCategory → 精准 nodeId
    ├─→ ASINRequestKeywordv2 → 反查核心关键词
    │
    ↓ 数据更精确、信息更全
    ↓ 可以做精确的财务分析（正向 P&L 48字段）
    ↓
    Step 1: 市场发现
    │   ├─ 写入「产品初选」表
    │   ├─ 写入「AI分析参考」表（29+字段）
    │   └─ ⛔ 弹出运营数据输入窗口
    │
    Step 2: Market Decision（市场可行性，确定性脚本）
    │   ├─ score_go_nogo.py → 加权评分
    │   └─ Hard Gates 检查
    │
    ├── GO(≥7.5) / CONDITIONAL(≥6.0) → 继续
    │   HOLD(4.0-5.9) / NO-GO(<4.0) → 停止，输出报告
    │
    Step 3: 属性标注 + 交叉分析 + VOC
    │   ├─ 写入「产品初筛」表（GO/COND触发）
    │   └─ 写入「产品开发」表（GO/COND触发）
    │
    Step 4: Financial Decision（财务可行性，decision.js）
    │   ├─ 缺成本 → 反向测算 → 输出采购价上限
    │   └─ 有成本 → 正向 P&L → 写入「财务分析」表（仅Financial GO）
    │
    Step 5: Launch Feasibility（资金可行性）
    │   └─ 3情景模拟（乐观/基准/悲观）
    │
    Step 6: 输出交付
        ├─ decision-card.html（速判卡）
        ├─ full-report.md（完整报告 ≥10章）
        └─ ⛔ 16项流程完整性检查
```

### 4.2 路径 B：从关键词进入（市场级）

```
关键词 (如 printer stand)
    │
    ├─→ KeywordRequest → 搜索量/CPC/关联类目
    ├─→ CategoryRequest(精准nodeId) → Top100
    ├─→ 从 Top100 选取参考 ASIN（可选）
    │
    ↓ 市场级分析，缺参考 ASIN 的精确售价/FBA
    ↓ 只能做反向财务（成本上限）
    ↓
    Step 1: 市场发现 → 写入初选+AI分析
    Step 2: Market Decision
    Step 3: 属性标注+VOC
    Step 4: ❌ 无法做正向P&L → 反向财务：输出采购价上限
    Step 5: ❌ 无需做Launch
    Step 6: 输出报告
```

### 4.3 两条路径的差异对比

| 维度 | ASIN 入口 | 关键词入口 |
|------|:--:|:--:|
| 市场分析 | ✅ | ✅ |
| 属性标注+交叉 | ✅ | ✅ |
| Go/No-Go | ✅ 精确 | ✅ 市场级 |
| **正向财务（P&L 48字段）** | ✅ | ❌ 需用户提供 ASIN |
| **反向财务（成本上限）** | ✅ | ✅ |
| 初筛 | ✅ GO/COND | ✅ GO/COND |
| 产品开发 | ✅ | ✅（细分市场级） |

## 5. 三层决策体系

### 5.1 核心概念

```
Market Decision（市场可行性）    ← 市场好不好？
        ↓
Financial Decision（财务可行性）  ← 能不能赚钱？
        ↓
Launch Feasibility（资金可行性）  ← 现金撑不撑得住？
        ↓
Overall Decision（综合决策）      ← 三个都过才过
```

**任何一层失败，整体不得为 GO。** 防止"市场好但亏钱"的错误。

### 5.2 Market Decision（市场可行性）

**计算方式**：`scripts/score_go_nogo.py` — 基于 22 维阈值表 + CLI 原始数据的确定性评分。

| 维度 | 权重 | 评分依据 | CLI 数据源 |
|------|:---:|---------|---------|
| 市场规模 | 25% | 搜索量 + 销量 YoY 增长率 | KeywordRequest + CategoryTrend(0) |
| 竞争格局 | 31.25% | ClickShare + 品牌数 + 新品率 + 评论 + CPC | KeywordRequest + CategoryRequest |
| 需求清晰度 | 18.75% | 搜索转化率（购买意图强度） | KeywordRequest |
| 进入壁垒 | 25% | 新品率 + 评论门槛 + CPC | CategoryRequest |

| 分数 | 决策 | 含义 |
|:---:|------|------|
| ≥7.5 | Market GO | 市场结构优秀 |
| 6.0-7.4 | Market CONDITIONAL | 市场可进入但需注意 |
| 4.0-5.9 | Market HOLD | 市场有结构性缺陷 |
| <4.0 | Market NO-GO | 不建议进入 |

### 5.3 Financial Decision（财务可行性）

**计算方式**：`scripts/financial_model/decision.js` — M值 + 净利率双重判定。

```
M = 实际回款金额 / 落地成本
落地成本 = 采购价 + 头程运费
```

| M值 | 净利率 | 决策 | 含义 |
|:---:|:---:|------|------|
| ≥2.0 | ≥15% | Financial GO | 每投入$1收回≥$2，利润≥15% |
| ≥1.6 | ≥10% | Financial CONDITIONAL | 可放量但需关注成本 |
| ≥1.3 | ≥0% | Financial HOLD | 勉强存活，利润安全垫薄 |
| <1.3 | — | Financial NO-GO | 亏本或无法收回成本 |

**反向财务**（缺成本时）：
```
"M ≥ 2.0 需要采购价 ≤ $X"
```
只在 CLI 有 CPC/CVR 数据时生成 ad-accurate 反向成本。

### 5.4 Hard Gates（一票否决项）

| Gate | fail 条件 | 影响 |
|------|---------|------|
| 专利 | 核心功能被有效专利保护且无法规避 | fail → 整体 NO-GO |
| 合规 | 需强制认证（FDA/FCC等）且无法获取 | fail → 整体 NO-GO |
| 产品安全 | 存在已知安全隐患 | fail → 整体 NO-GO |
| 供应链 | MOQ 过高或供应商不可得 | fail → 整体 NO-GO |

pending → 整体最多 CONDITIONAL GO

### 5.5 Launch Feasibility（资金可行性）

**计算方式**：`scripts/financial_model/dynamic.js` — 3 情景模拟。

| 情景 | 条件 |
|------|------|
| 乐观 | 销量 × 1.3 |
| 基准 | 用户提供的日销计划 |
| 悲观 | 销量 × 0.7 |

输出：峰值资金需求、回本月份、悲观情景回本。

### 5.6 三层决策 → AI-推荐推进映射

| Market | Financial | Launch | Hard Gates | AI-推荐推进 |
|:--:|:--:|:--:|:--:|------|
| GO | GO | GO | 全部 pass | **GO-推荐进入** |
| GO | COND | PENDING | pass/pending | **CONDITIONAL GO-有条件** |
| COND | GO | PENDING | pass/pending | **CONDITIONAL GO-有条件** |
| COND | HOLD | PENDING | pass | **HOLD-暂缓** |
| 任意 | NO-GO | 任意 | 任意 | **NO-GO-放弃** |
| 任意 | 任意 | 任意 | fail | **NO-GO-放弃** |

## 6. 6 张飞书表的推进逻辑

| # | 表名 | table_id | 写入条件 | 写入内容 |
|---|------|----------|---------|---------|
| 1 | **产品初选** | tbljJBWDxaLerJsN | **始终写入** | ASIN/关键词/类目/价格带/推进状态 |
| 2 | **AI分析参考** | tblMeN1P5LvXsp2e | **始终写入** | 29+字段（23结构化+8文本+4运营） |
| 3 | **产品初筛** | tblPhc8dHOelKPPF | GO 或 CONDITIONAL GO | 11/16 AI填写 + 5待人工确认 |
| 4 | **财务分析** | tblllu3JJYrG5KGE | **仅 Financial GO** | 48字段 P&L + 模型版本 + 输入哈希 |
| 5 | **产品开发** | tbl3dtXNC3Emy38k | GO 或 CONDITIONAL GO | 12/18 AI填写 + 成本待确认 |
| 6 | **供应商管理** | tblu1Q70vnmsiBZ5 | 用户提供 1688 链接后 | 供应商名/链接/价格/MOQ/样品 |

### 重要规则

- **AI分析参考每次必写**，无论后续是否推进。它是数据沉淀的核心。
- **财务分析仅在 Financial GO 时写入**。CONDITIONAL/HOLD/NO-GO 不写估算数据。
- **产品初筛和产品开发并行触发**，都依赖同一个条件（GO/CONDITIONAL），不是串行依赖。
- **NO-GO/HOLD 时后续表不写入是正确行为**，不是流程遗漏。

## 8. AI分析参考表 — 字段详解

> 以细分市场为分析单元。35 个字段，按组排列。

### 8.1 标识组（4字段）

| 字段 | 类型 | 说明 |
|------|------|------|
| 细分市场名称 | Text | 精准市场名。子类目同质化高直接用子类目名 |

| 关联初选 | Link | 指向产品初选表 |
| AI-推荐推进 | Select | GO/CONDITIONAL GO/HOLD/NO-GO/建议初筛/需更多数据 |

### 8.2 流量组（6字段）

| 字段 | 类型 | 说明 | CLI 来源 |
|------|------|------|---------|
| 核心词月搜索量 | Number | 核心关键词月搜索量 | KeywordRequest.SearchVolume |
| 核心词CPC | Number | 当前市场竞价(USD) | KeywordRequest.Cpc |
| 核心词搜索转化率 | Number(％) | 搜索→购买转化率。⚠️ 不是点击CVR | KeywordRequest.SearchConversionRate |
| 搜索量趋势方向 | Select | 快速增长/稳定增长/平稳/增速放缓/下降 | CategoryTrend(0) |
| 竞品总数 | Number | 关键词维度竞争产品数 | KeywordRequest.ProductCount |
| Top100月销额 | Number | 子类目Top100月销售额合计(USD) | CategoryRequest聚合 |

### 8.3 竞争组（8字段）

| 字段 | 类型 | 说明 | CLI 来源 |
|------|------|------|---------|

| 品牌数量 | Number | Top100独立品牌数 | CategoryRequest.Brand去重 |
| 竞争热度评估 | Select | 蓝海/分散/中等/集中/垄断 | 综合判断 |
| 类目新品占比 | Number(％) | Top100中上架≤6月占比 | CategoryRequest.OnlineDays分桶 |
| 平均评论门槛 | Number | Top100平均评论数 | CategoryRequest.RatingsCount |
| FBA占比 | Number(％) | FBA发货占比 | CategoryRequest.IsFBA聚合 |
| 亚马逊自营占比 | Number(％) | Amazon自营销量占比 | CategoryTrend(9) |
| 新品友好度 | Select | 友好/中等/困难/非常困难 | 综合判断 |

### 8.4 趋势组（3字段）

| 字段 | 类型 | 说明 | CLI 来源 |

| 方式 | 操作 | AI 动作 |

| B | 在飞书 AI分析参考表中手动填写 | 下次对话 AI 回读发现有值 → 自动触发 |
| C | 分批给 | 缺哪个补哪个，补齐自动触发 |

4 个字段（1688/头程/CPC/CVR）**全有值** → 立即执行 Step 4 财务分析。不需要说"进入财务分析"。广告订单占比可选项，默认 50%。

## 12. 后验追踪与成功画像

### 12.1 为什么需要后验追踪

分析时只能判断"能不能做"。只有上线后才知道"真的成功了吗"。不追踪结果，就永远不知道"什么样的产品在我手里真的赚钱了"。

### 12.2 追踪方式

### 12.3 发现规律

积累 20+ 条记录后，用飞书视图交叉筛选：

```
筛选条件: 6个月后状态 = "成功(盈利)"
    → 看这些成功案例有什么共性？
    → M值都≥2.0？CPC都<$1？新品率都>15%？
    → 形成你的"成功画像"
```

这就是从"每次临时分析"升级到"基于历史数据的可复用判断框架"。

## 13. 常见问题

### 13.1 为什么分析完了没有初筛/财务/开发？

AI-推荐推进 = NO-GO 或 HOLD 时，后续表不写入是正确行为。只有 GO/CONDITIONAL GO 才触发。

### 13.2 如何触发财务分析？

在 AI分析参考表填入 4 个字段（1688/头程/CPC/CVR），或在对话中告诉 AI。

### 13.3 搜索转化率能当 CVR 用吗？

**不能。** SearchConversionRate ≠ 点击 CVR。用错了会导致 M 值偏高——可能漏判亏损产品。

### 13.4 CPC为什么有两个字段？

核心词CPC（市场参考值，KeywordRequest）vs 当前CPC（你的实际出价，广告后台）。财务分析用后者。

### 13.5 分析一次大概消耗多少 API？

快速模式约 25 次，完整模式约 60 次。查看余额：`sorftime whoami`。

### 13.6 从关键词进入和从 ASIN 进入有什么区别？

关键词只能做市场级分析+反向财务（成本上限）。ASIN可以做精确正向 P&L（48字段）。详见[第 4 章](#4-完整使用路径)。

### 13.7 百分比字段为什么显示不对？

飞书百分比格式字段写入值必须用 0-1 小数。0.28 → 28%。写 28 → 2800%。详见[第 8.8 节](#88-字段写值规则)。
